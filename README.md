# smart_geofence

A smart geofencing layer on top of [`native_geofence`](../native_geofence).

`native_geofence` provides reliable native geofencing (Android Geofencing API +
iOS region monitoring). On **Android**, OS geofencing can be noisy and can
silently drop events. `smart_geofence` adds a battery-conscious smart layer on top
-- **without** continuous tracking:

- **Proximity monitor** — a low-power (BALANCED, displacement-gated) location
  stream that wakes the app only when it's near a fence.
- **Passive no-power monitor** — listens for `PRIORITY_PASSIVE` fixes produced
  for other apps (20 min interval / 10 min fastest / 40 min max wait). A
  confident passive fix can queue the transition without spending a fresh GPS
  fix; an ambiguous near-fence passive fix can escalate to foreground GPS confirm.
- **Motion gate** — uses Activity Recognition to skip FencePulse-triggered
  foreground confirms when the device is confidently still.
- **GPS confirm** — when near a boundary, queues a foreground-service
  high-accuracy confirm to decide inside/outside (accuracy-aware dead-band +
  teleport guard), then sends the confirmed transition to smart_geofence's
  postprocessor, which shares native_geofence's de-dupe state before queueing
  callback work, so OS events and confirmed events never double-fire.
- **Reconcile watchdog** — periodic WorkManager self-heal by default, with
  optional AlarmManager scheduling that is exact when permitted and inexact
  when exact alarms are unavailable.
- **FencePulse (opt-in)** — a bounded alarm-only near-boundary pulse. It does
  not fetch GPS directly; when movement suggests a fresh fix is useful, it
  queues the same foreground GPS confirm service used by proximity wakes.

iOS is served directly by `native_geofence` (it's already strong there);
`smart_geofence`'s smart layers are Android-only and a no-op elsewhere.

On Android, OS geofence transitions still run through `native_geofence`;
`smart_geofence` registers one internal callback there and routes those final
events back to the original callback registered for each geofence. Smart-confirmed
transitions use smart_geofence's postprocessor, which shares native_geofence's
delivered-state store before queueing the original callback. This keeps OS and
smart-confirmed transitions de-duped together while allowing smart_geofence to
add pre-processing and post-processing around the native engine.

## Usage

```dart
import 'package:smart_geofence/smart_geofence.dart';

@pragma('vm:entry-point')
Future<void> geofenceCallback(GeofenceCallbackParams params) async {
  // params.event, params.geofences, params.location (now carries
  // accuracyMeters + isMock).
}

await SmartGeofenceManager.instance.initialize(
  config: const SmartGeofenceConfig(
    escalation: SmartGeofenceEscalationConfig(
      proximity: SmartGeofenceProximityConfig(radiusMeters: 1000),
      passive: SmartGeofencePassiveLocationConfig(enabled: true),
    ),
  ),
);

await SmartGeofenceManager.instance.createGeofence(
  Geofence(
    id: 'home',
    location: Location(latitude: 37.4220, longitude: -122.0841),
    radiusMeters: 150,
    triggers: {GeofenceEvent.enter, GeofenceEvent.exit},
    iosSettings: const IosGeofenceSettings(initialTrigger: true),
    androidSettings: const AndroidGeofenceSettings(
      initialTriggers: {GeofenceEvent.enter},
    ),
  ),
  geofenceCallback,
);

// On app resume / boot:
await SmartGeofenceManager.instance.reconcile();
```

## Configuration guide

Start with the defaults. They are tuned for a battery-conscious Android
geofencing app: native geofences remain the primary wake source, passive fixes
are reused when available, and active GPS is spent only for short near-boundary
confirms.

### Core location layers

| Option | Default | Suggested range | Notes |
| --- | --- | --- | --- |
| `escalation.proximity.radiusMeters` | `1000` | `500..2000` | Wider catches more uncertain crossings but spends more confirms. Smaller values reduce active GPS use. |
| `escalation.proximity.priority` | `balancedPowerAccuracy` | keep default | Avoid `highAccuracy` for the long-lived proximity monitor unless you accept much higher battery use. |
| `escalation.proximity.interval` | `2 min` | `1..5 min` | Faster values improve responsiveness but wake more often. |
| `escalation.proximity.fastestInterval` | `1 min` | `30 sec..interval` | Lower values can process bursts more quickly. |
| `escalation.proximity.maxWait` | `5 min` | `interval..10 min` | Larger values let Android batch updates. |
| `escalation.proximity.minDisplacementMeters` | `200` | `100..300` | Smaller for walking-heavy apps; larger for less background work. |
| `escalation.passive.enabled` | `true` | `true` for most apps | Reuses fixes generated by other apps without turning GPS on. |
| `escalation.passive.priority` | `passive` | keep `passive` | Any other priority makes this an active location request. |
| `escalation.passive.interval` | `20 min` | `10..60 min` | Passive updates are opportunistic; Android may deliver none if no other app is using location. |
| `escalation.passive.fastestInterval` | `10 min` | `5..30 min` | Lower values consume more of the free fixes produced by other apps. |
| `escalation.passive.maxWait` | `40 min` | `20..120 min` | Larger values favor batching over freshness. |
| `escalation.passive.ambiguousConfirmEnabled` | `true` | `true` or `false` | `true` spends one active confirm when a passive fix is near but uncertain. `false` keeps passive strictly no-power. |

### Confirm and classification

| Option | Default | Suggested range | Notes |
| --- | --- | --- | --- |
| `escalation.gpsConfirm.priority` | `highAccuracy` | `highAccuracy` | One-shot confirms should usually be precise. Using `passive` may never resolve unless another app requests location. |
| `escalation.gpsConfirm.timeout` | `8 sec` | `5..9 sec` | Bounds each foreground confirm so the service can stop promptly. |
| `foregroundServiceLaunchTimeout` | `10 sec` | `5..20 sec` | Watchdog timeout for foreground confirm promotion. Timeout retries through AlarmManager. |
| `foregroundServiceStartDelay` | `1 sec` | `0..2 sec` | Brief wait before starting the confirm service so burst requests share one launch. `0` starts immediately. |
| `foregroundServiceRetryDelays` | `250 ms, 2 sec, 15 sec, 45 sec, 2 min` | short bounded list | Retry delays for failed or stuck foreground-service starts. Alarms are exact when permitted and inexact otherwise. |
| `escalation.boundary.minMarginMeters` | `20` | `10..50` | Larger values reduce false positives near the edge but increase ambiguous results. |
| `escalation.boundary.marginRatio` | `0.10` | `0.05..0.20` | Radius-relative safety band around the fence edge. |
| `escalation.boundary.maxMarginRatio` | `0.50` | `0.25..0.50` | Prevents large margins from making small fences impossible to classify. |
| `escalation.teleportGuard.enabled` | `true` | keep `true` | Rejects non-mock fixes that imply impossible travel speed. |
| `escalation.teleportGuard.maxSpeedMetersPerSecond` | `70` | `40..100` | About 250 km/h by default. Raise for fast-vehicle use cases. |

### FencePulse

FencePulse is off by default. It is a light alarm pulse, not a foreground
service. Most apps should leave it disabled unless they need repeated
near-boundary follow-up after an uncertain wake. When a tick needs a fresh GPS
fix, it queues the same short foreground GPS confirm service declared by
`smart_geofence`.

| Option | Default | Suggested range | Notes |
| --- | --- | --- | --- |
| `fencePulse.enabled` | `false` | enable only when needed | Starts light follow-up alarm ticks after unresolved near-boundary wakes. |
| `fencePulse.duration` | `2 hours` | bounded windows only | The FencePulse window stops after this duration. |
| `fencePulse.interval` | `75 sec` | `60..180 sec` | Shorter values improve follow-up responsiveness but may enqueue more confirms. |
| `fencePulse.minInterval` | `30 sec` | keep default | Safety floor applied to `fencePulse.interval`. |
| `fencePulse.maxIdleTicks` | `5` | `2..10` | Number of stationary ticks before FencePulse stops. |
| `fencePulse.exactAlarmStartDelay` | `250 ms` | keep default | First-tick delay after a near-boundary wake. |
| `fencePulse.motionGate.stationaryTtl` | `15 min` | `5..30 min` | How long a STILL reading can suppress FencePulse-triggered foreground confirms. |
| `fencePulse.notification.*` | varies | app-specific | Foreground confirm notification/channel settings. FencePulse itself shows no notification. |

### Recovery and diagnostics

| Option | Default | Suggested range | Notes |
| --- | --- | --- | --- |
| `reconcile.scheduler` | `workManager` | `workManager` or `exactAlarm` | `exactAlarm` uses AlarmManager. It is exact when permitted and inexact otherwise. |
| `reconcile.interval` | `30 min` | `15..60 min` | WorkManager repeat period, or next AlarmManager delay after each run. |
| `reconcile.initialDelay` | `0` | `0..5 min` | First WorkManager delay or first AlarmManager delay. |
| `reconcile.flexInterval` | `0` | optional | WorkManager-only; non-zero flex lets Android batch the watchdog. |
| `reconcile.requiresBatteryNotLow` | `false` | `false` for reliability | WorkManager constraint; AlarmManager mode checks battery manually when the alarm fires. |
| `logging.fileEnabled` | `false` | support builds | Enables app-private native logs that your app can export or clear. |
| `logging.maxFileBytes` | `256 KiB` | `64 KiB..1 MiB` | Native layer clamps each log file between 16 KiB and 5 MiB. |

`SmartGeofenceReconcileScheduler.exactAlarm` does not force exact-alarm
permissions onto every app. If exact alarms are not permitted, the watchdog logs
and uses an inexact AlarmManager alarm instead. WorkManager is only used as the
fallback when AlarmManager scheduling itself is unavailable.

## App-controlled log files

Android diagnostics can be written to bounded app-private log files. File logging
is disabled by default. When enabled, both `smart_geofence` and `native_geofence`
write native diagnostics that your app can fetch, display, export, upload,
redact, or clear later.

```dart
await SmartGeofenceManager.instance.configure(
  const SmartGeofenceConfig(
    logging: SmartGeofenceLogConfig(
      fileEnabled: true,
      maxFileBytes: 512 * 1024,
    ),
  ),
);

final logs = await SmartGeofenceManager.instance.readLogFile();
await SmartGeofenceManager.instance.clearLogFile();
```

The files live in the app's private Android storage (`noBackupFilesDir`) and are
not shared unless your app chooses to share them. Logs are still written to
logcat regardless of file-log settings.

## Permissions (base)

`smart_geofence` deliberately does **not** declare location or activity-recognition
permissions, so you only inherit what you actually request. Add to your app's
`AndroidManifest.xml`:

```xml
<!-- Required: proximity/passive layers + geofencing. Request
     ACCESS_BACKGROUND_LOCATION at runtime; the Android smart layers skip if it
     isn't granted. -->
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_BACKGROUND_LOCATION" />

<!-- Optional: only useful with the opt-in FencePulse; enables its motion-gate
     battery optimization. The motion gate no-ops if this isn't granted. -->
<uses-permission android:name="android.permission.ACTIVITY_RECOGNITION" />
```

`smart_geofence` declares `RECEIVE_BOOT_COMPLETED` (for reboot recovery), its own
proximity / motion / boot receivers, **and native_geofence's geofence-event +
reboot receivers** — so OS geofences fire and re-register after reboot with no
extra setup. (If you previously wired native_geofence's receivers in your own
manifest, you can remove them.)

`smart_geofence` also declares a callback foreground service. Use
`SmartGeofenceBackgroundManager.instance.promoteToForeground()` inside a
geofence callback when the callback needs extra time or needs to access APIs
that are safer from a foreground service. It shares the same notification
channel, title, icon, and notification id configured under
`fencePulse.notification.*`, so callback promotion does not create a separate
native_geofence notification.

## Migrating from `native_geofence`

`SmartGeofenceManager` mirrors `NativeGeofenceManager`'s API, so migration is mostly
a find-and-replace.

**1. Dependency** — replace `native_geofence` with `smart_geofence` (it pulls in
`native_geofence` transitively):

```yaml
dependencies:
  # native_geofence: ^1.3.1   # remove
  smart_geofence: ^1.0.0        # add
```

**2. Import + manager class** — the only required code change:

```dart
- import 'package:native_geofence/native_geofence.dart';
+ import 'package:smart_geofence/smart_geofence.dart';

- NativeGeofenceManager.instance.createGeofence(geofence, callback);
+ SmartGeofenceManager.instance.createGeofence(geofence, callback);
```

Every method is identical: `initialize`, `createGeofence`, `removeGeofence`,
`removeGeofenceById`, `removeAllGeofences`, `getRegisteredGeofenceIds`,
`getRegisteredGeofences`, `reCreateAfterReboot`.

**3. Unchanged** — your `Geofence` / `Location` / settings models, your
`@pragma('vm:entry-point')` callback and its `GeofenceCallbackParams` signature,
and all iOS behavior and setup. (`params.location` now additionally carries
`accuracyMeters` + `isMock`.) Prefer `SmartGeofenceBackgroundManager`
(`promoteToForeground` / `demoteToBackground`, or the `startForeground` /
`stopForeground` aliases) for callback foreground work so Android uses
smart_geofence's shared notification. `NativeGeofenceBackgroundManager` remains
re-exported for migration compatibility, but it uses native_geofence's separate
foreground-service notification.

**4. Two small gotchas**
- If you name the callback type explicitly, `GeofenceCallback` → `SmartGeofenceCallback`
  (just passing a function needs no change).
- Permissions are no longer provided by the plugin — declare location (and
  optionally activity-recognition) in your app manifest per
  [Permissions](#permissions-base). Migrating apps already declare location for
  geofencing, so this is usually a no-op.
- On Android, re-register existing geofences through
  `SmartGeofenceManager.createGeofence` after migrating. `smart_geofence` stores
  the original callback handle in its mirror, registers one internal dispatcher
  with `native_geofence` for OS events, and uses the same original callback
  handle when smart-confirmed events are queued by the postprocessor. Geofences
  that were only registered by `native_geofence` before the migration keep their
  old direct callback handle until refreshed.

**5. New (optional)**
```dart
await SmartGeofenceManager.instance.initialize(config: const SmartGeofenceConfig(...));
await SmartGeofenceManager.instance.reconcile(); // on app resume
```

**Behavioral changes:** same events, but more reliable on Android (passive
checks + escalation catch OS-dropped crossings) and de-duplicated (no doubles).
Slightly more -- bounded, low-power -- background work than bare
`native_geofence`. iOS is unchanged.

## Android foreground services and FencePulse

`smart_geofence` declares a small location foreground service for active GPS
confirms. Proximity/passive receivers only classify cheap fixes and enqueue work;
the fresh `getCurrentLocation` confirm runs after the service has promoted to the
foreground. The service start path keeps requests in a persisted queue and can
retry through AlarmManager. Retries are exact when the host app already has
exact-alarm access and inexact otherwise.

Callback foreground promotion is separate from GPS confirm monitoring. It exists
only to keep callback work alive after a geofence event has already fired:

```dart
@pragma('vm:entry-point')
Future<void> geofenceCallback(GeofenceCallbackParams params) async {
  await SmartGeofenceBackgroundManager.instance.promoteToForeground();
  try {
    // Do longer callback work here.
  } finally {
    await SmartGeofenceBackgroundManager.instance.demoteToBackground();
  }
}
```

The callback service uses the same foreground notification settings as the
active GPS confirm service (`fencePulse.notification.*`).

FencePulse is **off by default** and is now alarm-only. `smart_geofence` declares
its FencePulse receiver itself, and FencePulse does not require
`SCHEDULE_EXACT_ALARM` or a FencePulse foreground service.

The default FencePulse tick is 75 seconds. Tune it with
`SmartGeofenceConfig(fencePulse: SmartGeofenceFencePulseConfig(interval: ...))`
if your app needs a different cadence.

Most geofencing apps do **not** need FencePulse: the core path already wakes on
OS geofence crossings and uses short foreground confirms near uncertain
boundaries.
